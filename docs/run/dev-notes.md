# run 開発メモ

## 実装上の判断

| 判断内容 | 理由 |
|----------|------|
| run を司令塔（controller）＋永続状態（RunState）として設計 | 戦闘/マップ/各ノードは個別機能エリアに委ね、run は横断責務（状態・進行・RNG・報酬・HUD）に集中させ、結合点を NodeResult 一本に絞る |
| 休憩所/ショップ/宝箱/イベントをサブ項目に分割 | いずれも独立した UI 画面を持ち、単体で実装・検証可能。分割ルールの原則（独立 UI は分割）に従う |
| 報酬画面は分割せず親に保持 | 報酬は「抽選→RunState反映」が controller の責務に内包される小さな単位で、RunState 更新と密結合。独立 UI だが分割の実益が薄い |
| RNG を系統別の決定的ストリームに分離 | 原作の確率再現とテスト可能性の両立。マップ/カード報酬/モンスター/ポーション等で消費順が混ざらないようにする |
| 第 1 幕の完走を最小目標に段階実装 | 全幕・真ボスまでは段階的に拡張。まずノード解決ループが一巡する縦の動線を通す |
