# content 概要

## 目的・背景

ゲームに登場する具体データ（カード・レリック・ポーション・敵・イベント）の定義を担う機能エリア。
`combat` がシステム（仕組み）を完全再現するのに対し、`content` は**そのシステム上で動く具体物**を提供する。本プロジェクトの方針として、全コンテンツの網羅は求めず、アイアンクラッドで通常プレイが成立する**コア集合**を再現対象として選定する。

「再現する仕組みは完全、再現するコンテンツ量は厳選」という役割分担の、コンテンツ側を担う。

## スコープ

### 作るもの

- **カード定義（アイアンクラッド + 共通）**：攻撃/スキル/パワー、コスト、効果、アップグレード、レアリティ。初期デッキ＋報酬で出る代表的なカード群
- **レリック定義**：初期レリック（バーニングブラッド）＋戦闘/取得フックを持つ代表的なレリック群
- **ポーション定義**：代表的なポーション（効果・対象）
- **敵定義**：第 1 幕の通常/エリート/ボスを中心に、HP・インテントパターン・行動 AI を持つ敵群
- **イベント定義**：代表的な `?` イベント（選択肢と結果）
- **エンカウンター定義**：マップのモンスター/エリート/ボスノードに割り当てる敵編成

各データは ID で参照され、`combat`・`run`・`map` から駆動される。combat 側の汎用フック（GameAction/AbstractPower）を組み合わせて各カード/敵の効果を表現する。

### 作らないもの

- 他キャラ（サイレント/ディフェクト/ウォッチャー）専用カード・レリック
- 全カード/全敵/全レリック/全ポーション/全イベントの網羅
- アセンション専用の調整値

## 制約

- `combat` のシステム（アクション/パワー/ダメージ計算）と `run`/`map` の ID 参照に依存。content 単独では動かず、combat の仕組みの上で表現する。
- 原作アセット（画像/数値表記）は同梱しない。数値・効果ロジックの再現を主とし、見た目はプレースホルダ/代替アセットで補う。
- 再現対象の選定基準を明確化し（後述）、段階的に増やせる構造（データ駆動 + レジストリ）にする。

## 完了条件

- 初期デッキと報酬で出るコアカード群が定義され、combat で正しく機能する
- 第 1 幕の代表的な敵編成（通常/エリート/ボス）が定義され、戦闘が成立する
- 初期レリックと代表ポーションが定義され、run/combat で機能する
- 各定義が ID レジストリ経由で参照でき、ユニットテストで効果を検証できる

## 再現対象の選定基準（初期）

- アイアンクラッドの初期デッキ（ストライク/ディフェンド/バッシュ）は必須
- 通常プレイで頻出かつシステム多様性を示すカード（ブロック獲得、状態異常付与、ドロー、パワー）を優先
- 第 1 幕の敵を中心に、インテント/AI の代表パターンを網羅できる編成を優先
- レリック/ポーション/イベントは「フック種類の代表」を優先（量より仕組みの網羅）

## サブ項目への分割

データ種別ごとに独立して定義・テスト可能なため分割する（`/add-feature content/<name>`）。

- `content/cards` … カード定義
- `content/relics` … レリック定義
- `content/potions` … ポーション定義
- `content/monsters` … 敵・インテント・AI・エンカウンター
- `content/events` … イベント定義
