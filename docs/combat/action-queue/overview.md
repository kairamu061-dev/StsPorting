# action-queue 概要

## 目的・背景

戦闘中の全処理を逐次実行するアクションキューを実装する。combat の中でも最優先・最重要のサブ項目であり、「システム/エフェクト完全再現」の根幹を成す。

原作 STS は、ダメージ・ブロック・ドロー・パワー付与・カード移動などのあらゆる効果を **GameAction** という単位でキューに積み、`GameActionManager` が 1 つずつ解決していく。効果が別の効果を誘発する場合は、実行中のアクションがキュー**先頭**に新たなアクションを割り込ませる。この「逐次解決＋割り込み」のモデルこそが、原作の処理順・演出タイミング・パワー相互作用を生み出している。

本サブ項目はこのモデルを忠実に再現し、他の全戦闘システム（カード/パワー/敵/エフェクト）が乗る土台を提供する。

## スコープ

### 作るもの

- **GameAction**：処理単位の抽象基底（`update()` で自身を進め、完了で `isDone`）
- **ActionManager**：キュー本体。「現在のアクション」を毎フレーム解決し、完了したら次へ
- **割り込み（addToTop）/ 末尾追加（addToBottom）**：実行中アクションが先頭/末尾へ別アクションを積む仕組み
- **解決ループの統合点**：CombatScreen の update から駆動され、キューが空になるまで処理を進める
- **カードプレイ要求（CardQueueItem）のキュー投入**：プレイヤー操作をアクション列へ変換する入口
- **基本 GameAction 群**：DamageAction / GainBlockAction / DrawAction / DiscardAction / ApplyPowerAction / ExhaustAction など、他サブ項目が共通利用する原始アクション
- **ターン制御フックの発火点**：ターン開始/終了・カードプレイ後など、定常処理アクションを差し込む口

### 作らないもの

- 個々のカード/パワー/敵の具体効果（cards/powers/enemy が GameAction を組み合わせて表現）
- 演出そのもの（effects。ただしアクションが演出開始の合図を出す連携点は持つ）
- ターン構造の上位ロジック（turn-flow。action-queue は解決機構を提供）

## 制約

- 描画から独立し、ヘッドレスで決定的にユニットテストできること（乱数は RunRng 経由）。
- 1 フレームに複数アクションが解決し得るが、演出を伴うアクションは完了まで次へ進まない（待ち時間の概念を持つ）。原作の「演出待ち」挙動を再現する。
- 割り込みの挿入順を厳密に定義し、同条件で常に同じ解決順になること（再現性）。

## 完了条件

- ActionManager が空になるまでキューを解決し、`addToTop`/`addToBottom` の順序が定義どおりになる
- DamageAction 等の基本アクションが CombatState を正しく更新する
- 「ダメージ → 被弾フックで追加効果が割り込み → 先に解決」のような割り込みシナリオがテストで再現できる
- 演出待ちアクション（duration を持つ）が指定時間だけ次の解決を保留する
- 上記がヘッドレスユニットテストで決定的に通る

## 画面イメージ

UI を持たない内部機構。解決の概念図：

```
[キュー]  front → [ A ][ B ][ C ] ← back
   1) current = A を取り出し update() を毎フレーム実行
   2) A が実行中に addToTop(X) → front → [ X ][ B ][ C ]
   3) A.isDone になったら current = X（割り込みが先に解決）
   4) 以降 B, C ... 空になるまで継続
```
