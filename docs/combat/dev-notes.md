# combat 開発メモ

## 実装上の判断

| 判断内容 | 理由 |
|----------|------|
| 原作のアクションキュー（GameActionManager）方式を踏襲 | 効果・パワー誘発・演出の**順序とタイミング**こそが「システム/エフェクト完全再現」の根幹。即時計算ではなく逐次解決にすることで原作挙動を一致させる |
| ロジック（CombatState/Action）と描画（CombatScreen/vfx）を分離 | 戦闘ロジックをヘッドレスで決定的にユニットテストできるようにする |
| マスターデッキはコピーで戦闘に投入 | 戦闘中の消滅/生成を戦闘内に閉じ、永続デッキ変更は run 側（報酬/ショップ/休憩）に一元化する |
| combat を 7 サブ項目に分割（action-queue 等） | それぞれ独立システムで単体実装・検証が可能。分割ルールに従う。action-queue は全システムの土台のため最優先 |
| ダメージ計算順を明文化（攻撃修正→被ダメ修正→ブロック→HP→被弾フック） | 筋力/弱体/脆弱/ブロック/とげの相互作用を原作と一致させるため、順序を設計で固定 |

## 今後の課題

- ボス固有ギミック（特殊勝利条件・行動制限）は段階対応。まず通常戦闘の縦通しを優先
- レリックの戦闘内フック（ターン頭/被弾時/カードプレイ時など）の発火点を action-queue 設計確定時に整理
- 入力・演出の体感一致は Windows 実機での反復確認が必要

## ユーザへの要望

- 原作の細かな数値・確率（敵 AI の連続行動制限、報酬金額レンジ等）は再現対象を絞る前提。特に「ここは数値まで完全一致させたい」敵/カードがあれば優先指定してほしい
